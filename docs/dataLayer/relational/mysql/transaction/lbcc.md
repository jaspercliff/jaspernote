# lbcc 

LBCC（Lock-Based Concurrency Control，基于锁的并发控制）

虽然 MVCC 完美解决了读写并发问题，但对于 **“写-写”冲突** 或者需要 **“实时读取最新数据（当前读）”** 的场景，必须依赖锁机制（LBCC）来确保绝对的数据一致性。

---

### 1. 锁的属性分类（共享锁与排他锁）

按照对数据的排他程度，锁可以分为：

* **共享锁（S 锁 / Share Lock）**：又称读锁。多个事务可以同时持有同数据的 S 锁。
* 手动加锁方式：`SELECT ... LOCK IN SHARE MODE;`（MySQL 8.0 推荐 `FOR SHARE`）


* **排他锁（X 锁 / Exclusive Lock）**：又称写锁。若某事务持有了 X 锁，其他事务既不能加 S 锁，也不能加 X 锁，必须阻塞等待。
* 手动加锁/隐式触发：`UPDATE`、`DELETE`、`INSERT` 或 `SELECT ... FOR UPDATE;`



---

### 2. 锁的粒度与算法（InnoDB 的三种行锁形态）

InnoDB 的行锁本质上是**加在索引项上的**（如果表没有索引，会直接退化为锁表！）。在 **RR 隔离级别** 下，行锁有三种具体实现算法：

```text
       间隙 (Gap)       记录 (Record)      间隙 (Gap)
     ( ... , 10 )          [ 10 ]        ( 10 , 20 )
 ──────────────────────┬─────────────┬─────────────────►
                       │             │
                       │             └─ Next-Key Lock (10, 20]
                       └─────────────── Record Lock [10]

```

#### ① 记录锁（Record Lock）

* **作用**：仅仅锁住**单条具体的索引记录**。
* **场景**：在主键或唯一索引上进行**等值精确匹配**查询并加锁时（如 `WHERE id = 10 FOR UPDATE`），间隙锁会降级为记录锁。

#### ② 间隙锁（Gap Lock）

* **作用**：锁住两个索引记录之间的**开区间 `(A, B)**`，或者第一条记录之前/最后一条记录之后的间隙。**防止其他事务在该范围内插入新记录**。
* **主要目的**：**专门用于防止“幻读”**。
* **特点**：间隙锁之间不互斥。两个事务可以同时在同一个间隙上加 Gap Lock。

#### ③ 临键锁（Next-Key Lock）

* **构成**：**记录锁 (Record Lock) + 间隙锁 (Gap Lock)** 的结合体。
* **范围**：**左开右闭区间 `(A, B]**`。
* **地位**：**InnoDB 在 RR 隔离级别下的默认行锁算法**。
* 任何一条记录的 Next-Key Lock，管辖的范围永远是“它自己”加上“它左边的间隙” 但在 10 的左边，永远趴着一个系统内置的极小值 Infimum(-∞)

#### use 

- 默认看范围或非唯一索引 👉 统统上 临键锁（Next-Key Lock）。
- 主键/唯一索引 + 精确查 + 能找到 👉 降级为 行锁（Record Lock）（只锁一行）。
- 主键/唯一索引 + 精确查 + 找不到 👉 降级为 间隙锁（Gap Lock）（只锁缝隙，防插入）

### 3. LBCC 如何解决“当前读”下的幻读问题？

在 RR 级别下：

* **快照读（普通 `SELECT`）**：依赖 **MVCC** 解决幻读（通过 Read View 过滤掉新插入的数据）。
* **当前读（`SELECT ... FOR UPDATE` / `UPDATE` 等）**：依赖 **LBCC（Next-Key Lock）** 解决幻读。

**举个例子：**
假设数据库有一表，索引列有值 `10` 和 `20`。

1. 事务 A 执行：`SELECT * FROM t WHERE id >= 10 FOR UPDATE;`
2. InnoDB 会在范围索引上施加 **Next-Key Lock**，锁定范围为 `( -∞, 10 ]`、`( 10, 20 ]` 以及 `( 20, +∞ )`。
3. 此时事务 B 尝试执行 `INSERT INTO t VALUES (15)`，试图插入 `15`。
4. 由于 `15` 落在被事务 A 锁定的 `(10, 20]` 间隙内，事务 B **直接被阻塞**，直到事务 A 提交。
5. 这样就通过锁硬生生阻塞了“新数据的插入”，彻底杜绝了当前读场景下的幻读现象。

# 第三部分：MVCC 与 LBCC 的对比与协同

这两者并不是对立的，而是**互相配合**构成完整的数据库并发事务引擎。

| 比较维度 | MVCC（多版本并发控制） | LBCC（基于锁的并发控制） |
| --- | --- | --- |
| **解决场景** | 读 - 写并发 | 写 - 写并发、强一致的“读 - 写”并发 |
| **核心机制** | Undo Log 版本链 + Read View | 共享/排他锁 + 记录锁/间隙锁/临键锁 |
| **触发方式** | 普通 `SELECT`（隐式快照读） | `INSERT/UPDATE/DELETE` 或 `SELECT ... FOR UPDATE`（当前读） |
| **并发性能** | **极高**（无锁，读写互不阻塞） | **较低**（涉及锁竞争与阻塞） |
| **数据时效性** | 读历史快照（取决于 Read View 生成时机） | 读最新数据（实时最新状态） |

### 场景串联总结：

* **普通 `SELECT`（快照读）**：走 **MVCC**。不加锁，根据 Read View 顺着 Undo Log 找旧版本，性能极高。
* **带锁 `SELECT` / 增删改（当前读）**：走 **LBCC**。先把最新的记录锁住（Next-Key Lock），确保数据不会在修改过程中被其他事务破坏或插入，然后再执行读取或变更操作。
