# mvcc 

在 MySQL（特别是 InnoDB 存储引擎）中，为了在“高并发”和“数据一致性”之间取得平衡，设计了两套核心机制来处理并发事务：**MVCC（多版本并发控制）** 和 **LBCC（基于锁的并发控制）**。

简单来说：

* **MVCC 负责“读-写”并发**：让读操作不阻塞写操作，写操作也不阻塞读操作，通过“数据历史快照”实现高效读取（即**快照读**）。
* **LBCC 负责“写-写”或“强一致读”并发**：通过对数据加锁，强制让并发修改或特定查询排队执行，保证数据的绝对安全（即**当前读**）。


---

##  MVCC（Multi-Version Concurrency Control，多版本并发控制）

MVCC 的核心思想是：**“写数据时保留旧版本，读数据时根据规则挑选合适的数据版本”**。

它主要由三大组件协作完成：**隐藏列**、**Undo Log 版本链** 和 **Read View（一致性视图）**。

---

### 1. 三大底层组件

#### ① 记录中的隐藏列（Record Hidden Columns）

InnoDB 存储引擎在每行数据的真实列之外，都会隐式地添加三个隐藏字段：

| 隐藏字段 | 含义 | 作用 |
| --- | --- | --- |
| **`DB_TRX_ID`** | 事务 ID | 记录**最后一次修改/插入**该行记录的事务 ID（单调递增）。 |
| **`DB_ROLL_PTR`** | 回滚指针 | 指向该行记录在 **Undo Log** 中的上个版本的地址。 |
| **`DB_ROW_ID`** | 隐藏主键 | 如果表没有指定主键或唯一索引，InnoDB 会自动生成这个自增 ID 作为主键。 |

#### ② Undo Log（回滚日志）与版本链

当事务对某行记录进行 `UPDATE` 或 `DELETE` 时：

1. 并不直接覆盖旧数据，而是先把当前旧数据拷贝一份写入 **Undo Log**。
2. 将当前行记录的 `DB_TRX_ID` 修改为当前事务的 ID。
3. 将当前行记录的 `DB_ROLL_PTR` 指向 Undo Log 中刚写入的旧版本。

随着多次修改，这些 Undo Log 记录通过 `DB_ROLL_PTR` 指针串联起来，就形成了一条从最新数据到最旧数据的 **版本链**：

```text
【当前表最新记录】
age = 25 | DB_TRX_ID = 300 | DB_ROLL_PTR ──┐
                                          │
    ┌─────────────────────────────────────┘
    ▼
【Undo Log 版本 2】
age = 22 | DB_TRX_ID = 200 | DB_ROLL_PTR ──┐
                                          │
    ┌─────────────────────────────────────┘
    ▼
【Undo Log 版本 1】
age = 18 | DB_TRX_ID = 100 | DB_ROLL_PTR ──> NULL

```

#### ③ Read View（一致性视图）

当事务执行 `SELECT` 时，MVCC 会生成一个 **Read View**。它相当于当时数据库状态的一个“快照切片”，里面包含了以下 4 个关键属性：

* `m_ids`：生成 Read View 时，当前系统中正在活跃（已开启但尚未提交）的事务 ID 列表。
* `min_trx_id`：`m_ids` 中的最小值。
* `max_trx_id`：系统准备分配给**下一个新事务的 ID**（即当前已出现的事务 ID 最大值 + 1）。
* `creator_trx_id`：创建该 Read View 的当前事务本身的 ID。

---

### 2. Read View 的版本可见性判断规则

当某个事务读取一行数据时，它会拿着数据版本链中的 `DB_TRX_ID`，依次对照当前 Read View 进行如下判断：

```text
                  min_trx_id           max_trx_id
                       │                   │
  已提交的事务         │   活跃事务(m_ids)  │     未来开启的事务
  (肯定可见)          │   (需进一步比对)   │      (肯定不可见)
───────────────────────┴───────────────────┴───────────────────────► 事务ID (DB_TRX_ID)

```

1. **`DB_TRX_ID == creator_trx_id`**：修改该版本的事务就是自己，**可见**。
2. **`DB_TRX_ID < min_trx_id`**：说明修改该版本的事务在生成 Read View 之前就已经提交了，**可见**。
3. **`DB_TRX_ID >= max_trx_id`**：说明修改该版本的事务是在生成 Read View 之后才开启的，**不可见**。
4. **`min_trx_id <= DB_TRX_ID < max_trx_id`**：
* 如果 `DB_TRX_ID` **在 `m_ids` 列表中**：说明生成 Read View 时该事务还未提交，**不可见**。
* 如果 `DB_TRX_ID` **不在 `m_ids` 列表中**：说明生成 Read View 时该事务已经提交了，**可见**。



**若某个版本被判定为“不可见”，则顺着 `DB_ROLL_PTR` 顺藤摸瓜找 Undo Log 中的上一个版本，重复上述判断，直到找到第一个可见的版本为止。**

---

### 3. 不同隔离级别下的 MVCC 区别

MVCC 只在 **RC（读已提交）** 和 **RR（可重复读）** 隔离级别下生效：

* **RC（Read Committed）**：**每次 `SELECT` 语句执行时，都会重新生成一个新的 Read View**。
* *结果*：能读到最新已提交的数据（解决了脏读），但同一个事务内两次查询 Read View 不同，会导致读到的数据不一致（存在不可重复读）。


* **RR（Repeatable Read）**：**事务内第一次 `SELECT` 时生成 Read View，整个事务生命周期内一直复用它**。
* *结果*：无论其他事务怎么修改提交，看到的始终是第一次查询时刻的数据状态（解决了脏读和不可重复读）。
