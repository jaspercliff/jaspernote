# Distro 协议

如果说 **JRaft** 是 Nacos 用来保证数据绝对不能错（强一致性 CP 模式）**的内核，那么 **Distro** 就是 Nacos 用来保证系统**绝对不能挂、吞吐量极高（高可用性 AP 模式）的自研杀手锏。

---

## 1. 什么是 Distro 协议？

**Distro 是 Nacos 专门为“非持久化服务发现（Ephemeral Service）”场景自研的一种 AP（可用性/分区容错）分布式共识协议。**

在微服务架构中，普通的业务微服务实例（比如你的订单服务、用户服务部署了 100 个节点）具有以下特点：

1. **高频变化**：服务经常扩容、缩容、重启，心跳极其频繁。
2. **允许短暂不一致**：某个实例刚挂掉，其他 Nacos 节点延迟一两秒才感知到，并不会导致灾难性后果（大不了重试一下）。
3. **极度追求高可用**：哪怕 Nacos 集群断网、分裂了，每个 Nacos 节点也必须能正常接待业务微服务，绝对不能拒绝服务。

Distro 协议就是为了这种场景而生的。它不需要像 Raft 那样搞复杂的投票选主，也不需要每做一件事都得超过半数节点同意。

---

## 2. Distro 的核心工作原理（大白话拆解）

Distro 协议的核心思想可以概括为八个字：**“人人平等，数据对齐”**。

### ① 路由分片（责任到人）

在 Distro 协议里，Nacos 集群的每个节点都是平等的，没有 Leader 和 Follower 之分。
但是，为了防止混乱，它们会把注册上来的微服务实例按照 **哈希算法（Hash）** 进行分片。

> 比如有 3 个 Nacos 节点（A、B、C）。当“订单服务”启动并向 Nacos 注册时，通过计算 Hash，它被分配给了 **Nacos A**。那么，Nacos A 就是这个订单服务实例的 **“负责节点”（Owner）**。

### ② 异步广播（事后通报）

* **写操作**：订单服务直接把心跳和注册请求发给 Nacos A，Nacos A 在本地内存写完后，**立刻**给订单服务返回成功（响应极快）。
* **同步**：随后，Nacos A 会在后台启动一个**异步任务**，把这个新注册的实例信息广播（Sync）给 Nacos B 和 Nacos C。

### ③ 最终一致性（定期对账）

既然是异步广播，万一因为网络抖动，Nacos B 没收到 A 发过来的同步数据怎么办？
Distro 引入了**定期对账机制**：

* 每个 Nacos 节点会定期向其他节点发送一份自己负责的数据的 **轻量级指纹（MD5 校验和）**。
* Nacos B 收到 A 的 MD5 后，比对一下发现和自己本地的不一样，说明自己漏掉了一些数据。
* 此时 Nacos B 会主动向 Nacos A 发起请求：*“兄弟，我数据落后了，把订单服务的最新明细全量拉给我一份。”* 从而实现**最终一致性**。

---

## 3. Distro 面对极端情况的表现

### 如果某个 Nacos 节点挂了怎么办？

假设负责订单服务的 Nacos A 彻底宕机了。由于之前的异步同步，Nacos B 和 C 的内存里其实已经有了订单服务的备份数据，所以微服务依然可以正常从 B 和 C 获取服务列表，**业务完全不受影响**。
同时，原本连在 A 上的微服务客户端发现 A 挂了，会自动重连到 B 或 C。B 或 C 接收到新的长连接后，会根据哈希重新计算，自动接管（Become Owner）这些实例，继续维持它们的心跳。

---

## 4. Distro vs JRaft（Nacos 内部的双核驱动）

在 Nacos 内部，这两个协议是完美互补的：

| 维度 | Distro 协议 (AP) | JRaft 协议 (CP) |
| --- | --- | --- |
| **应用场景** | **临时服务实例**（默认的微服务注册） | **持久化服务实例、配置中心数据、元数据** |
| **数据存储** | 纯内存（缓存），挂了靠心跳重建 | RocksDB / 数据库（必须落盘） |
| **一致性** | 最终一致性（异步对账） | 强一致性（Raft 日志复制） |
| **性能** | 极高（单机写内存，无网络阻塞） | 中等（需要等待集群多数派确认） |
| **默认端口** | `8848`（Nacos 2.x 集群间同步） | `7848`（专用 Raft 选主与同步） |

### 总结

**Distro 协议是 Nacos 能抗住百万级微服务实例高频上下线、心跳轰炸的核心秘密。** 它摒弃了传统的强一致性锁缚，用“哈希分片 + 异步同步 + 定期对账”的轻量级玩法，换取了极致的性能与高可用。
