# g1 

## 标记复制 
而 G1 把内存切成了两千多个 Region：

在 Young GC 时：G1 选定所有的年轻代 Region，把里面存活的对象，复制到全新的、干净的 Survivor Region 或 Old Region 中。

在 Mixed GC 时：G1 根据停顿时间目标，选定全部年轻代 Region + 收益最高的几个老年代 Region 组成一个回收集（Collection Set，简称 CSet）。然后同样把这些 Region 里存活的对象，统一复制到另外几个全新的、干净的空闲 Region 中。


一句话概括：

G1（Garbage First）= 面向大内存、低停顿、可预测停顿时间的垃圾回收器。

从 JDK 9 开始，它就是 HotSpot JVM 的默认垃圾回收器。 

--- 

## 最大的特点——Region

以前的 JVM：

| Young |      Old         |

Young、Old 都是一整块连续空间。

而 G1：

| R1 | R2 | R3 | R4 | R5 |  
| R6 | R7 | R8 | R9 |R10 |

整个堆被切成很多 Region（区域）。

每个 Region 大小相同：

1MB
2MB
4MB
8MB
16MB
32MB

JVM 根据堆大小自动选择 Region 大小，目标通常是不超过约 2048 个 Region。 

--- 

Region 不是固定身份

例如：

开始：

R1 Eden
R2 Eden
R3 Survivor
R4 Old
R5 Old

GC 后：

R1 Old
R2 Empty
R3 Eden
R4 Old
R5 Survivor

可以看到：

Region 的身份是动态变化的。

不像以前：

Young 永远是 Young
Old 永远是 Old

--- 

## 为什么叫 Garbage First？

假设：

Region1
100MB
垃圾90MB
存活10MB

Region2
100MB
垃圾20MB
存活80MB

Region3
100MB
垃圾95MB
存活5MB


先回收 Region3 Region1

因为：

复制存活对象最少
释放空间最多

所以叫：

Garbage First（垃圾优先）

G1 会优先选择垃圾比例最高的 Region 回收，而不是一次扫描整个老年代。 

--- 

## G1 的三种 GC

实际上只有三种。

- Young GC
- Mixed GC
- Full GC


### ① Young GC

和以前 Minor GC 类似。

例如：

Eden满了

触发：

Young GC

回收：

Eden + Survivor

存活对象：

复制
↓
新的 Survivor
或者
Old

因为是复制算法：

没有碎片


### ② Mixed GC（G1 最大特点）

这是 G1 最重要的地方。

CMS：

Old 满了 -> CMS ->  整个老年代扫描

而 G1：

Young + 垃圾最多的 Old Region

一起回收。

例如：

Young：
R1
R2
Old：
R10
R15
R22
垃圾特别多

一次 Mixed GC：

回收
R1
R2
R10
R15

不会回收：

所有 Old

所以：

停顿时间可控。


### ③ Full GC

最不希望发生。

什么时候发生？

例如：

复制对象时：

Old
没有地方放了

或者：

Evacuation Failure

只能：

Full GC

Stop The World。

所以：

G1 设计目标就是：

尽量不要 Full GC。



## G1 的整体流程


### 第一步：Young GC

对象：new 进入：Eden Region

满了： Young GC


### 第二步：Concurrent Mark（并发标记）

什么时候开始？

例如：

Old 使用率
达到阈值

开始：

Concurrent Mark

特点：

和用户线程一起运行

不是 STW。

作用： 统计：哪个 Region 垃圾最多


### 第三步：Remark

因为：

Concurrent Mark：

用户线程
一直在修改引用

所以：

最后：

Remark
Stop The World

修正最终结果。

时间一般较短。 

### 第四步：Cleanup

统计：哪些 Region 值得回收

建立：Collection Set(CSet)

例如：

R8垃圾95% 加入CSet


### 第五步：Mixed GC

开始真正回收。

例如：

Young
+
Old
里面
垃圾最多
几个 Region

复制：

活对象
↓
新的 Region

旧 Region：

整个释放

## Remembered Set（RSet）

这是 G1 最核心的数据结构之一。

问题：

假设：

Region1引用Region1000

如果回收：Region1000难道扫描整个堆？

所以：每个 Region 都维护：Remembered Set

记录：有哪些 Region引用了我

例如：

R10
RSet：
R2
R8
R50

说明：

R2
R8
R50
里面有对象
引用了 R10

这样：

扫描：RSet即可
不用扫描整个堆，大大降低了回收成本。 


## 停顿时间为什么可预测？

G1 有一个目标参数：

-XX:MaxGCPauseMillis=200

意思不是：

GC 一定在 200ms 内完成。

而是：

G1 会根据历史统计，尽量选择合适数量的 Region，使停顿时间接近这个目标。

例如：

如果估算：

回收10个Region
100ms

那就回收 10 个。

如果：

回收30个Region
400ms

那可能只回收一部分，剩余留到下一次 Mixed GC。

这是 G1 停顿时间可预测的重要原因。 

## G1 优缺点

优点：

* Region 化管理，回收粒度更细
* 停顿时间更可预测
* 支持大堆（GB 甚至几十 GB）
* 并发标记，减少长时间 STW
* 复制整理，几乎没有内存碎片
* 默认垃圾回收器（JDK 9+）

缺点：

* 实现复杂，维护 RSet 等结构有额外开销
* 在小堆场景下，吞吐量未必优于 Parallel GC
* 极端情况下仍可能发生 Full GC
